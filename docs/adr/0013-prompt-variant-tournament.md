# ADR-0013: Prompt Variant Tournament — Online Evaluation and Automated Prompt Optimization

| Field | Value |
|---|---|
| **Status** | Accepted |
| **Date** | 2026-04-05 |

## Context

### The prompt optimisation ceiling of SkillBank

ADR-0006 (SkillBank) injects extracted strategies into the system prompt. ADR-0007 (OMLS) trains model weights on successful trajectories. Both systems improve _what_ an agent knows — but neither systematically improves _how_ the agent's base prompt is written.

Prompt wording has outsized impact on LLM behaviour. The same underlying capability can be expressed in a dozen structurally different ways — concise vs. verbose, chain-of-thought vs. direct, example-heavy vs. principle-driven — and the best formulation varies by model, task distribution, and workload. Manual A/B testing of prompt variants is impractical at scale.

The tournament system adds a third learning loop that operates at the prompt level, above SkillBank and below OMLS:

| Loop | Layer | Timescale | Infrastructure |
|---|---|---|---|
| SkillBank (ADR-0006) | Skill injection | Immediate | None |
| **Tournament (ADR-0013)** | **System prompt** | **Per-release** | **Free model CI** |
| OMLS (ADR-0007) | Model weights | Weekly (idle) | GPU |

### Seed agents and the variant problem

Orager ships six built-in seed agents: `explorer`, `planner`, `researcher`, `coder`, `reviewer`, `tester`. Each has a handwritten system prompt. These prompts were written once and never systematically evaluated against alternatives.

To improve them, we need:
1. A set of prompt rewriting strategies to try
2. A benchmark harness to measure performance per variant
3. A production mechanism to select better variants at runtime
4. A promotion gate that prevents premature changes

### Vision model tracking

A parallel problem exists for vision-capable models: orager's vision tasks (screenshot analysis, diagram interpretation) run against whichever model is configured globally. There is no mechanism to track which vision model produces the best outputs on vision-specific benchmarks.

## Decision

### 1. Prompt variant generation (11 strategies)

`src/agents/refine.ts` generates 12 variants per agent (original + 11 rewrites):

| Strategy ID | Description |
|---|---|
| `original` | Baseline — unchanged prompt |
| `concise` | Strip filler, tighten phrasing |
| `structured` | Add explicit section headers |
| `chain_of_thought` | Add CoT reasoning instruction |
| `output_format` | Specify output structure explicitly |
| `examples` | Prepend 1-2 worked examples |
| `relax` | Remove hedge language and uncertainty |
| `strict` | Add explicit constraint list |
| `role_emphasis` | Strengthen role framing |
| `step_by_step` | Decompose task into numbered steps |
| `self_check` | Add self-verification instruction |
| `meta_prompt` | Ask model to reason about best approach |

Each variant receives a stable `variantId` hash so scores accumulate across runs.

### 2. Benchmark task corpus

`src/agents/benchmark.ts` defines static benchmark tasks with objective `verify()` functions — deterministic checks that don't require an LLM judge. Example: "How many exported functions are in `src/agents/score.ts`?" verified by checking that the output includes the function names.

`src/agents/generated-tasks/` holds synthetic tasks generated by `scripts/generate-tasks.ts` using a free model. The corpus is expanded automatically by the `generate-tasks.yml` workflow (PR 6) and currently contains ~150 tasks across 5 agent types.

Generated tasks use `successCriteria` strings (natural language) rather than code `verify()` functions, and are scored by the LLM judge.

### 3. Tournament harness

`tests/prompt-tournament.test.ts` orchestrates the evaluation:

1. Load all seed agents
2. For each agent, generate all 12 prompt variants
3. For each benchmark task, pick a variant deterministically via seeded RNG (`taskIndex + SEED_OFFSET`)
4. Run the task via `runAgentLoop` against a rotated pool of free models
5. Score via `verify()` (static tasks) or judge score (generated tasks)
6. Record to `agents.sqlite` via `recordAgentScore()`
7. After all tasks, call `getBestVariant()` and log results

**SEED_OFFSET** allows multiple passes to cover different variant/task combinations without repeating work.

**TOURNAMENT_MODELS** rotates across free models per-task to deconfound model effects from variant effects.

### 4. LLM-as-judge semantic scoring

`src/agents/judge.ts` implements a judge that scores any agent output against a `successCriteria` string. The judge:

- Assigns a `score` in [0, 1]
- Returns `pass: boolean` (threshold 0.7)
- Reports `confident: boolean` (low-confidence judgments are down-weighted)
- Returns a `reason` string for debugging

Judge scoring is optional (enabled via `JUDGE_ENABLED=1`) and supplements — never replaces — the `verify()` function for static tasks. For generated tasks, a `judgePass=true` overrides the soft length check.

Judge scores are stored alongside binary success in `agents.sqlite` for use in composite scoring (ADR-0013 §6).

### 5. Vision model benchmark track

`tests/vision-tournament.test.ts` runs a parallel tournament for vision-capable models. Five vision tasks (screenshot analysis, chart interpretation, diagram mapping, etc.) run against each model in `VISION_MODELS`. Results are stored per `modelId` in the same `agents.sqlite` database and surfaced in the tournament dashboard.

### 6. Confidence-weighted promotion (Wilson score + composite)

`getBestVariant()` in `src/agents/refine.ts` uses a confidence-weighted comparison rather than raw win rate:

**Wilson lower bound** (Bayesian 95% CI lower bound for a binomial proportion):

```
wilson(p, n) = (p + z²/2n - z·√((p(1-p) + z²/4n)/n)) / (1 + z²/n)
```

where `z = 1.96` (95% confidence), `p = successes/n`.

This naturally penalises variants with few runs — a variant that went 5/5 on 5 tasks gets a Wilson score of ~0.57, not 1.0.

**Composite score** blends Wilson with judge scores:

```
composite(successes, n, judgeAvg) = 0.7 × wilson(successes, n) + 0.3 × judgeAvg
```

The judge weight drops to zero when no judge data is available (pure Wilson fallback).

**Promotion gates:**
- `PROMOTION_MIN_RUNS = 5` — both the original and challenger must have ≥5 runs
- `PROMOTION_MARGIN = 0.05` — challenger composite score must exceed original by ≥0.05

### 7. Production epsilon-greedy deployment

When `ORAGER_RECORD_SCORES=1`, `orager run` and `orager chat` deploy variants to real user sessions:

```
80% exploit → getBestVariant() winner (or original if no winner yet)
20% explore → assignVariant() random selection
```

This ensures all variants continue to receive traffic (preventing convergence on a local optimum) while directing most traffic to the current best.

For `orager chat`, the variant is selected once per session and held constant across all turns (session pinning). This prevents confounding from within-session variant switching.

Scores are recorded as fire-and-forget via `recordProductionScore()` — failures are silently swallowed to prevent score recording from affecting agent execution.

### 8. Promoted variants file

`src/agents/promoted-variants.json` stores the current winner per agent. When a variant clears the promotion gates, `scripts/tournament-promote.ts` writes the winning `variantId` and strategy to this file. The production epsilon-greedy uses the promoted variant as its exploit target.

Promotion is automated: the burn-in workflow (`tournament-burnin.yml`) calls `tournament-promote.ts` after each run and opens a PR if winners are found.

### 9. Tournament dashboard UI

`orager serve` exposes `GET /api/tournament` returning:

```json
{
  "agents": [
    {
      "agentId": "explorer",
      "variants": [
        {
          "variantId": "explorer-v1-concise",
          "strategy": "concise",
          "runs": 42,
          "successRate": 0.81,
          "vsBaseline": 0.12,
          "avgTurns": 3.1,
          "avgCostUsd": 0.0031,
          "avgJudgeScore": 0.78
        }
      ]
    }
  ],
  "visionModels": [...]
}
```

`ui/src/tabs/Tournament.tsx` renders a live leaderboard with agent filter, 60s auto-refresh, win-rate badges, and a vision model section. The tab is accessible at `/tournament` in the browser UI.

### 10. Burn-in workflow

`.github/workflows/tournament-burnin.yml` runs on manual dispatch:

- 2–3 passes with different `SEED_OFFSET` values
- `continue-on-error: true` on each pass — variant failures record scores and execution continues
- Vision passes after text passes
- Scores persist via GitHub Actions cache (`agents-db-<run_id>`) across passes
- `tournament-summary.ts` prints the leaderboard on completion
- `tournament-promote.ts` runs with `if: always()` — winners are crowned even when some variants fail
- A promotion PR is opened automatically if any variant clears the gates

## Consequences

### Positive

- **Systematic prompt improvement**: Every seed agent now has a data-driven path to a better system prompt, revisited every burn-in cycle.
- **Free**: The tournament uses free-tier models (`meta-llama/llama-3.3-70b-instruct:free`, `deepseek/deepseek-chat-v3-0324:free`, `qwen/qwen3.6-plus:free`). A full 2-pass run costs $0.
- **Non-blocking**: Score recording is fire-and-forget. The tournament does not affect agent execution latency.
- **Composable with OMLS**: Tournament variants improve prompts; OMLS trains weights. The two loops are orthogonal and mutually reinforcing.

### Negative / trade-offs

- **Slow convergence**: With `PROMOTION_MIN_RUNS=5` per variant and 12 variants per agent, crowning a winner requires ~60+ task runs per agent. A single burn-in pass covers ~5 tasks per agent; 6+ passes are needed to saturate all variants.
- **Free model variance**: Free models vary in capability and availability. Results from `llama-3.3-70b` may not generalise to `claude-3-5-sonnet`. The tournament is a relative signal, not an absolute one.
- **Static benchmark drift**: `benchmark.ts` tasks reference specific files (`src/agents/score.ts`, `src/types.ts`). If these files change significantly, `verify()` functions may need updating.
